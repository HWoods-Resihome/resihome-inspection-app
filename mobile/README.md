# ResiWALK Mobile — Capacitor Native Shell (Stage 0)

This is the **native iOS/Android shell** for ResiWALK. It lives in the `mobile/`
subfolder of the web app repo so you have **one repo for both**, but it has its
own build/deploy lifecycle (Xcode / Android Studio) that is completely separate
from the Vercel web deploy.

> **Vercel ignores this folder.** The repo root has a `.vercelignore` listing
> `mobile/`, so the web app's Vercel build never touches the native shell. Run
> all `npm`/`cap` commands below from *inside* `mobile/`, not the repo root.

## What this is
Stage 0 of the Path B roadmap (see `PATH_B_ANALYSIS.md` in the web repo). The
shell loads the **live web app** (on Vercel) inside a native webview. There are
**no web-app logic changes** and **no bundled web build** — the website stays
exactly as deployed. This exists to get ResiWALK into the App Store / Play Store
and validate native packaging, camera permissions, and store review before any
offline work.

## Files
- `capacitor.config.ts` — points the webview at the live URL (`server.url`).
  Switch to `https://resiwalk.com` after Safe Browsing clears.
- `www/index.html` — offline fallback shown only if the device can't reach the
  server at launch (Stage 0 has no real offline yet).
- `OAUTH_WEBVIEW.md` — the one real integration risk and its fix (Google blocks
  OAuth in embedded webviews; sign-in opens in the system browser + deep-link
  return). **Read before testing login.**
- `native-oauth-bootstrap.ts` — reference implementation of the OAuth bridge.
- `BUILD_RUNBOOK.md` — step-by-step to initialize, build, and submit to both
  stores. **Run on a Mac.**

## Quick start (on your Mac)
```bash
npm install
npx cap init "ResiWALK" "com.resihome.resiwalk" --web-dir=www
npx cap add ios && npx cap add android
npx cap sync
npx cap open ios      # build & run in Xcode
npx cap open android  # build & run in Android Studio
```
Then follow `BUILD_RUNBOOK.md` for permissions, the OAuth URL scheme, and store
submission.

## Important
- The native project folders (`ios/`, `android/`) are generated by
  `npx cap add` on your machine — they are not in this scaffold.
- Test login FIRST (OAuth-in-webview is the main risk). See `OAUTH_WEBVIEW.md`.
- The web app is untouched by Stage 0. All offline/SQLite/sync work is Stage 2+.
